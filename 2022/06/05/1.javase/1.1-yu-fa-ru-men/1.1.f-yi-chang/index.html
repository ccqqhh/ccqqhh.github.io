15.1 异常分类

内容导视：

  • RuntimeException
  • Exception

程序的运行并不总是一帆风顺的，可能读取的文件不存在、读取的不是个文件、使用值为 null 的引用访问实例变量、访问不存在的下标、强行将字符串转成整数...这些不正常的情况称为异常。（语法错误不是异常）

运行时发生异常（抛出了异常对象），如果不处理，默认退出程序，异常后面的代码不会执行。

int i = 5 / 0;
System.out.println("后面的代码不再执行...");
Exception in thread "main" java.lang.ArithmeticException: / by zero
	at com.cqh.Excep.main(Test.java:1)

所有异常类都是 Throwable 的子类，体系图：

15.1.1 Error

Error：JVM 无法解决的严重错误，如系统内部错误、资源耗尽等错误，如： StackOverflowError（栈溢出）和 OutOfMemoryError（内存溢出）；程序会崩溃。

应用程序不应该抛出这种类型的对象。 如果出现了这样的内部错误，除了通告给用户，并尽力使程序安全地终止之外， 再也无能为力了。

这种情况很少出现，大多数情况下我们关注于 Exception。

如果线程请求的栈深度大于虚拟机所允许的深度，将会抛出 StackOverflowError 异常；如果虚拟机栈可以动态扩展时却无法申请到足够的内存，就会抛出 OutOfMemoryError 异常。

15.1.2 RuntimeException

Exception：因为逻辑错误或偶然的外在因素导致的一般性问题，可以使用针对性的代码处理。

Exception 分为两大类：编译时异常、运行时异常。

运行时异常又称非受检异常，不要求强制处理；RuntimeException（包括子类）是运行时异常。

通常是由于程序逻辑错误导致的异常（内因），如空指针、除 0、下标越界、类型转换、数字格式化等运行时异常。这类异常如果强制要求处理，比如计算、调用方法、访问字段、访问数组某个下标的元素等，都需要 try catch，那么代码量会急速膨胀，看着就很混乱。

这类异常从一开始就不该存在，应极力避免，可以提早检查出来；使用 try catch 处理这类异常有些小题大作，效率也低。

例：

obj.method();

应考虑到 obj 为 null 的情况，避免空指针异常：（如果能确定 obj 一定不为 null，则不需要处理）

if (obj != null) {
    obj.method();
}

不要使用 try catch：（无法通过预检查的异常除外，比如 NumberFormatException）

try {
    obj.method();
} catch (NullPointerException e) {
    e.printStackTrace();
}

常见的运行时异常

  1. java.lang.NullPointerException 空指针异常

    Object obj = null;
obj.getClass();

  2. java.lang.ArithmeticException 算数异常

    public static void main(String[] args) {
    double num = div(5, 0);
}
private static double div(int dividend, int divisor) {
    return dividend / divisor;
}

  3. java.lang.ArrayIndexOutOfBoundsException 数组下标越界异常

    int[] arr = {6, 2, 7};
System.out.println(arr[9]);

  4. java.lang.ClassCastException 类型转换异常

    Object obj = new ArrayList();
String str = (String)obj;

  5. java.lang.NumberFormatException 数字格式化异常

    Integer i = Integer.valueOf("不是数字");

15.1.3 Exception

除 RuntimeException 外的 Exception 是编译时异常，又称受检异常，必须手动处理，否则无法通过编译。

FileInputStream fis = new FileInputStream("E:\\a.txt");
java: 未报告的异常错误java.io.FileNotFoundException; 必须对其进行捕获或声明以便抛出

这类异常，并不是程序本身逻辑有问题，而是取决于外部环境，并不取决于代码，如 E 盘下的 a.txt 是否存在。

我们无法控制这些外在因素，为此需要手动处理 FileNotFoundException，说明当此文件不存在时应该怎么办。

try {
    FileInputStream fis = new FileInputStream("E:\\a.txt");
} catch(FileNotFoundException e) {
    System.out.println("a.txt 不存在，我们应该...");
}

15.2 异常处理

内容导视：

  • throws
  • try catch

对于编译时异常必须处理，否则通不过编译；处理异常的方式有两种，throws 上抛给调用者，当前方法终止运行；或是 try catch 处理异常，程序继续运行。

15.2.1 throws

当方法中的语句可能抛出某个异常时，但是不确定如何处理这种异常，则此方法应显示地声明抛出异常，表明该方法将不对这些异常进行处理，而由该方法的调用者负责处理：try catch 或继续上抛。

在方法声明中用 throws 语句声明抛出异常的列表，throws 后面的异常类型可以是方法中可能抛出的异常类型，也可以是它的父类。如把 FileNotFoundException 换成 Exception，即它的父类。

语法：

方法声明 throws 异常1, 异常2 ... {}

表明此方法可能抛出 FileNotFoundException 异常

public static void some() throws FileNotFoundException {
    // 此处可能会有异常抛出，当这句抛出异常时，后面的代码不会执行
    FileInputStream fis = new FileInputStream("E:\\a.txt");
    ...
}

调用者必须处理此编译时异常，是继续上抛，还是 try catch 自己处理。

public static void main(String[] args) throws FileNotFoundException {
    // 此处可能会有异常抛出，当这句抛出异常后，后面的代码不会执行
    some();
    ...
}

throws 过程：

  • 发生异常后，异常后的代码不会执行，直接结束方法回到调用者处，由调用者决定如何处理此异常

  • 如果 main 方法自己不处理异常，而是继续上抛给 JVM，异常后的代码不会执行；JVM 会调用异常对象的 printStackTrace 方法打印出异常信息，然后退出程序。

运行时异常默认以 throws RuntimeException 的方式处理（隐式）。

public void some() throws RuntimeException {
    int a = 5 / 0;
}
// 等同于
public void some() {
    int a = 5 / 0;
}

子类重写父类的方法时，对抛出异常的规定：子类重写的方法，所抛出的异常类型要么和父类抛出的异常一致，要么为父类抛出的异常类型的子类型（或者干脆不显示抛出异常）。这是为了多态，父类出现的地方可以使用子类替换，且不会出现异常或错误。

class Animal {
    public void run() {}
}
// 任意子类
class ？ extends Animal {
    // 假如这里可以通过编译，不受限制地抛出异常
    public void run() throws Exception {}
}
class Test {
    // 这里不确定传入哪个子类对象
    public static void run(Animal a) {
        /*
        	编译时，看编译类型的此方法没有抛出异常，正常通过；
        	运行时，调用的是子类重写后的方法，但子类重写后的方法如果抛出了编译时异常
        	但这里又没有处理，或者告知此方法可能会抛出异常（throws 或 try catch）
			相当于绕过了编译器的检查，开发者很可能就漏掉了对此异常的处理，导致程序崩溃
        */
        a.run();
    }
}

所以经常看到父类的方法声明抛出异常，而实际没有抛出异常，强行要求处理，是预见到重写后的方法很有可能抛出异常。

15.2.2 try catch

try 包含可能出错的代码，当异常发生时由对应的 catch 处理。

语法：

try {
	// 可能会出现异常的代码 
} catch (异常类型 变量名) {
	// 发生异常后应执行的代码 
}

执行 try catch 语句过程：

  • 执行 try {} 中的语句
    • 如果没有抛出异常，catch {} 中语句不会执行（直至 try 中语句执行结束，整个 try catch 结束，程序继续运行）
    • 如果抛出了异常，异常处后面的语句不会执行，直接执行对应 catch 中的语句（直至 catch 语句执行结束，整个 try catch 语句结束，程序继续运行）
Random random = new Random();
int b = random.nextInt(2);
try {
    int a = 3;
    System.out.println("中");
    int c = a / b;
    System.out.println("后");
} catch (ArithmeticException e) {
    System.out.println("前");
}

当 b != 0 时，顺利执行完 try 中的代码，输出 中、后；

当 b = 0 时，输出 中，执行到 int c = a / b; 这行抛出异常，跳转至对应 catch 块，输出 前。

异常并不是越早处理越好，当调用者已做好准备，执行某段代码出现异常情况后该如何处理，结果调用的方法内部自己 try catch 了异常，没有上抛，但其实根本没有处理好，也没有打印异常栈轨迹信息，外部调用者没有接收到抛出的异常，以为程序还正常运行，没有启动应急方案，这些无意被忽略的异常也许会酿成大祸，直到很久后才被发现。

我无意就犯这样的错，有时候爆红强迫我处理异常，我快捷键随便一按，默认 try catch 处理，便编写其它地方的代码，正好埋下了隐患，但察觉不了，程序还能继续运行，直到要使用数据时，才发现不对劲，疑惑不是已经写了 catch 处理吗？debug 一步步调试，原来被内部方法自己处理掉了，外部的 catch 没有生效，此时程序都不知道跑了多远了...

15.2.3 多个 catch 块

当 try 包围的代码内抛出了不止一个异常，需要多个 catch 子句，用于捕获不同的异常，要求：父类异常必须在子类异常的后面、无法捕捉不可能抛出的编译时异常。

发生异常后，从上至下选择对应的 catch 子句执行，只会执行一条 catch 分支 。（重复捕捉没有意义）

try {
    FileInputStream fis = new FileInputStream("E:/a.txt");
    int i = 5 / 0;
} catch (FileNotFoundException e) {
    System.out.println("文件没有找到");
} catch (RuntimeException e) {
    System.out.println("出现了运行时异常");
} catch (Exception e) {
    System.out.println("出现了其它编译时异常");
}

如果把父类 Exception 放在前面，发生的子类型异常一定会被此 catch 捕捉到，后面捕捉子类型异常 FileNotFoundException 的 catch 永远不会执行，没有意义。

没有捕捉到的运行时异常会抛给上一级调用者；不捕捉的编译时异常必须以 throws 的方式处理。

try {
    int b = 5 / 0;
} catch (NullPointerException e) {
    // 打印异常信息
    e.printStackTrace();
}

如果多个异常同一种处理方式，除了使用父类型的异常囊括统一处理，还可以在 catch 中将不同类型的异常以 | 分隔，统一处理。（被分隔的异常不得有父子关系）

JDK7 新增特性：multi-catch 多重捕获机制

try {
    int i = 5 / 0;
    FileInputStream fis = new FileInputStream("E:/a.txt");
    DriverManager.getConnection(null);
} catch (RuntimeException | FileNotFoundException | SQLException e) {
    System.out.println("异常已发生");
}

15.2.4 finally

始终会执行的代码可以放入 finally 中；

// 如果将 fis 定义在 try 内，则 finally 内无法访问到此变量
FileInputStream fis = null;
try {
    fis = new FileInputStream("E:\\a.txt");
} catch (FileNotFoundException e) {
    e.printStackTrace();
} finally {
    // 如果 fis 不等于 null，就调用 close 方法关闭资源
    if (fis != null) {
        try {
            // 此方法也会抛出异常，所以需要 try catch 处理
            fis.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

执行 try 语句时，不管异常是否发生，都会执行 finally 中的代码（释放资源、关闭连接），除非退出程序或被打断。

强迫处理异常的确非常繁琐，当确定文件一定存在时，本应是如下语句，现在倒是变得不那么简单清晰明了。

FileInputStream fis = new FileInputStream("E:\\a.txt");
fis.close();

如果没有出现异常，则执行 try 块中所有语句，不执行 catch 块中语句；如果有 finally，最后还需要执行 finally 里面的语句。

如果出现异常，则 try 块中异常发生后，剩下的语句不再执行。将执行 catch 块中的语句；如果有 finally，最后还需要执行 finally 里面的语句。

有人就问了，就算异常发生，异常都被 catch 捕捉了，有没有 finally 不都会继续执行吗？

FileInputStream fis = null;
try {
    fis = new FileInputStream("E:\\a.txt");
} catch (FileNotFoundException e) {
    e.printStackTrace();
} 
// 应继续执行
if (fis != null) {// 避免空指针异常
    try {
        fis.close();
    } catch (IOException e) {
        e.printStackTrace();
    }
}

不一定，如下例执行到 return; 语句，整个方法就结束了，不会执行剩下的语句。

try {
    int i = 5;
    return;
} catch (RuntimeException e) {
    e.printStackTrace();
}

System.out.println("请执行我");

但是如果放入 finally 中：

try {
    int i = 5;
    return;
} finally {
    System.out.println("请执行我");
}

在 return 前，会先执行 finally 中代码，输出 请执行我。

try finally 配合使用，这种用法相当于没有捕获异常，需要上抛给调用者处理。不管是否发生异常，都必须执行某段代码。

15.2.5 try with resources

JDK7 新增特性，用于简化关闭流的过程；不需要在 finally 块中关闭流了，try 语句结束后会自动调用对象的 close 方法。（若 try 中抛出了异常，try 剩余代码不再执行，在进入 catch 之前先关闭流）

// 在 try () 中放入需要关闭资源的对象（实现了 AutoCloseable 接口的实现类对象）
// 可以放置多个对象，以 ; 分隔
try (FileInputStream fis =  new FileInputStream("E:\\a.txt")) {
} catch (FileNotFoundException e) {
    e.printStackTrace();
} catch (IOException e) {
    // 关闭流（调用 close 方法）时如果抛出了异常，由此 catch 块处理
    e.printStackTrace();
}

JDK7 需要在 try 括号中需要声明变量，如 try (FileInputStream fis = f），JDK9 后可以直接放入，如 try (f）。

如果不想处理 try 中抛出的异常，但需要关闭资源：

public static void some() throws FileNotFoundException {
    FileInputStream f = null;
    try {
        f = new FileInputStream("E:\\a.txt");
    } finally {
        try (FileInputStream f1 = f) {
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

也不想处理 close 方法抛出的异常：

public static void some() throws IOException {
    try (FileInputStream f = new FileInputStream("E:\\a.txt")) {
        ...
    }
}

或者抛出异常，但不想强制调用者必须处理异常，可以转为运行时异常：

public static void some() throws IOException {
    try (FileInputStream f = new FileInputStream("E:\\a.txt")) {
        ...
    } catch (IOException e) {
        throw new RuntimeException("invoke failed
", e);
    }
}

调用 close 方法的顺序

class Test {
    public static void main(String[] args) {
        try (MyStream m1 = new MyStream("s1");
             MyStream m2 = new MyStream("s2");
             MyStream m3 = new MyStream("s3")) {
            throw new Exception("run");
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
class MyStream implements AutoCloseable {
    private String name;

    public MyStream(String name) {
        this.name = name;
        System.out.println("creating " + name);
    }

    @Override
    public void close() throws Exception {
        /*if (true) {
            throw new Exception(name);
        }*/
        System.out.println("closing" + name);

}

运行：

creating s2
creating s3
closings3
closings2
closings1

	at com.cqh.cover.Test.main(OverrideException.java:22)

与声明顺序相反，逆序调用对象的 close 方法。

期间某个对象调用构造器时抛出了异常，不再创建剩下的对象，逆序调用已创建好的对象的 close 方法，try 语句块结束。

使得创建 m3 对象时抛出异常：

try (MyStream m1 = new MyStream("s1");
     MyStream m2 = new MyStream("s2");
     MyStream m3 = new MyStream("s3");
     MyStream m4 = new MyStream("s4")) {
    System.out.println("try 中的语句不执行");
} catch (Exception e) {
    System.out.println(e);
}

运行：

creating s1
creating s2
creating s3
closings2
closings1
java.lang.RuntimeException: s3 constructor failed

将 close 方法的注释打开，注释 try 中抛出的异常。

运行：

不调用 addSuppressed 方法添加被抑制的异常

始终会执行的 finally 块有时候会覆盖掉 try 中的异常：

try {
	throw new RuntimeException("try：throw Exception"); 
} finally {
    throw new RuntimeException("finally：throw Exception"); 
}

运行：

// finally 中抛出的异常
Exception in thread "main" java.lang.RuntimeException: finally：throw Exception
	at com.cqh.cover.Test.main(OverrideException.java:22)

try 抛出的异常还没有处理，接着又抛出了一个异常，try 异常信息丢失，只记录下了 “finally：throw Exception”。

改进

RuntimeException re = null;
try {
    throw new RuntimeException("try：throw Exception"); 
} catch (RuntimeException e) {
    re = e;
    // 抛出 try 中的异常
    throw re;
} finally {
    try {
        throw new RuntimeException("finally：throw Exception"); 
    } catch (RuntimeException e) {
        // 如果 try 中没有抛出异常，直接抛出 finally 内的异常
        if (re == null) {
            throw e;
        } else {
            // 否则 try 抛出的异常添加 e 到被抑制的异常列表中
            re.addSuppressed(e);
        }
    }
}

运行：

// try 中抛出的异常
Exception in thread "main" java.lang.RuntimeException: try：throw Exception
	at com.cqh.cover.Test.main(OverrideException.java:21)
    
    // finally 中被抑制的异常
	Suppressed: java.lang.RuntimeException: finally：throw Exception
		at com.cqh.cover.Test.main(OverrideException.java:27)

15.3 自定义异常

内容导视：

  • 抛出自定义异常
  • 常用方法

15.3.1 步骤

当程序出现了某些错误，如果内置的异常类无法满足需求，可以自定义异常，描述此错误信息。

步骤：

  1. 定义类，编译时异常继承 Exception，运行时异常继承 RuntimeException

  2. 调用父类的有参构造，以传递错误信息

public class UserNameFormatException extends Exception {
    public UserNameFormatException(String message) {
        super(message);
    }
}

15.3.2 抛出异常

在你认为发生错误的地方抛出此异常对象，throw new 异常类("错误原因");

如果抛出的是编译时异常，需要在方法处使用 throws 声明可能会发生的异常。

public User getUserByName(String name) throws UserNameFormatException {
    if ("xxx".equals(name)) {
        throw new UserNameFormatException("用户名称非法！");
    }
    ...
}

throw 语句只能放在 {} 中的最后一句，抛出异常后，方法直接结束，返回上一级，由调用者处理此异常。

15.3.3 常用方法

构造器(String message, Throwable cause)：错误信息，异常发生的原因。当捕获了异常后，再次抛出新异常时（目的是改变异常类型，比如当前方法不支持抛出编译时异常、忽略细节等原因），如果不希望丢失原始异常的信息，使用此构造；或者调用 initCause 方法。

try {
    // 可能会抛出的异常
    ...
} catch (FileNotFoundException e) {
    // 抛出此异常的原因是 try 中抛出了 FileNotFoundException
    throw new IOException("读写文件失败！", e);
}

void addSuppressed(Throwable exception)：添加被抑制的异常。

如果由调用者处理异常；在 try 块中抛出了异常后，在返回至调用者前，会执行 finally 块中的语句；但是 finally 块在执行的过程中，也可能会抛出异常。

如果 finally 块也抛出了异常，则此异常被返回给调用者，但之前 try 块中的异常就丢失了。

如下例，try 抛出的算术异常，就被空指针异常覆盖了，控制台上只会打印空指针异常。

try {
    int i = 5 / 0;
} finally {
    String str = null;
    str.length();
}

同理，不要在 finally 中使用 return 语句，会忽略 try 中的异常或返回语句。

try {
    throw new Exception("被忽略的异常");
} finally {
    return;
}

在构造器中也是如此，

当 try 中出现的异常才是问题的根源，可以忽略 finally 中的异常（抑制此异常）。

如下例，但是感觉还是挺麻烦：

// 准备抛出的异常
RuntimeException cue = null;
try {
    int i = 5 / 0;
} catch (ArithmeticException a) {
	cue = a;
} finally {
    try {
        String str = null;
    	str.length();
    } catch (NullPointerException n) {
        /*
        只有当准备抛出的异常为 null 时（说明 try 中没有异常）
        才记录 finally 中的异常
        */
        if (cue == null) {
            cue = n;
        }
    }
    // 抛出异常，由调用者处理
    if (cue != null) {
        throw cue;
    }
}
RuntimeException cue = null;
try {
    int i = 5 / 0;
} catch (ArithmeticException a) {
	cue = a;
    throw cue;
} finally {
    try {
        String str = null;
    	str.length();
    } catch (NullPointerException n) {
        // 只有 cue 为 null 时才抛出 finally 内的异常
        if (cue == null) {
            throw n;
        }
    }
}

如果不想丢失异常信息，使用 addSuppressed 方法，添加被抑制的异常。

RuntimeException cue = null;
try {
    int i = 5 / 0;
} catch (ArithmeticException a) {
	cue = a;
} finally {
    try {
        String str = null;
    	str.length();
    } catch (NullPointerException n) {
        if (cue == null) {
            cue = n;
        } else {
            // 添加被抑制的异常
            cue.addSuppressed(n);
        }
    }
    if (cue != null) {
        throw cue;
    }
}
RuntimeException cue = null;
try {
    int i = 5 / 0;
} catch (ArithmeticException a) {
    cue = a;
    throw cue;
} finally {
    try {
        String str = null;
        str.length();
    } catch (NullPointerException n) {
        if (cue == null) {
            throw n;
        } else {
            cue.addSuppressed(n);
        }
    }
}

使用 try-with-resources 时，当 try 中抛出异常，自动调用 close 方法也抛出了异常，会抑制 close 方法抛出的异常，同时调用 addSuppressed 添加至原有异常。

Throwable getCause()：获取原始异常（本异常发生的原因）

String getMessage()：获取错误信息

StackTraceElement[] getStackTrace()：获取栈轨迹元素数组，可以据此定位异常位置；默认开启，可以关闭此功能；

在创建异常对象时，调用如下构造器，writableStackTrace 赋值为 false，就不会调用 fillInStackTrace 方法记录栈轨迹，即有关当前线程的栈帧的当前状态的信息；此时调用 getStackTrace 方法，返回的数组没有元素。

构造器(String message, Throwable cause,
                    boolean enableSuppression,
                    boolean writableStackTrace)

栈轨迹（stack trace）是一个方法调用过程的列表，记录了抛出异常的方法以及所在行数。构造器中默认调用 fillInStackTrace 记录栈轨迹，可以手动调用此方法更新异常抛出点的调用栈信息，将之前的信息丢弃。

15.4.1 单元测试

编写了多个方法，我需要验证它们的可行性；于是编写一个测试类，最开始在 main 方法多次调用方法1，传入不同的参数，使用 sout 将结果打印在控制台上，运行程序，肉眼辨别结果是否是预期值；不是则在方法1多处插入 sout 语句，观察究竟是哪里出了问题，或者 debug；好，发现错误处修改后，再手动将 sout 语句一个个删掉。

好，方法1验证完毕，因为只有一个入口方法，只能将 main 方法中的测试代码清空，继续测试方法2、方法3...

这时突然想起测试方法1时，忽略了某处细节，又将 main 方法清空，凭着回忆复原测试代码，如此反复...

的确很麻烦，不是吗？使用单元测试，每个方法都可单独运行，不需要在 main 方法内反复修改。

IDEA 在 Project Structure 的模块下添加 Dependencies，导入 junit-4.13.1、hamcrest-all-1.3 的 jar 包，jar 包可以在 Maven 仓库中搜索并下载：https://mvnrepository.com/

然后创建 src 的同级目录 test，右键此目录 Mark Directory as/Test Sources Root，使之变绿；

然后在 test 目录下编写测试类，包名与被测试类所在包一致，在方法上添加注解 @Test；（import org.junit.Test;）

如果是 Maven 项目，测试类放在 src/test/java 下。

想要运行哪个方法就点击绿色三角（或 Ctrl + Shift + F10），点击类左边的三角运行所有方法。

规则：

  • 公开的类
  • 公开的实例方法，没有返回值

规范：

  • 类名为 测试的类名 + Test
  • 方法名为 测试的方法名 + Test

下面是常见的测试注解：

@Test：把一个方法标记为测试方法
@Before：每一个测试方法执行前自动调用一次
@After：每一个测试方法执行完自动调用一次
@BeforeClass：所有测试方法执行前执行一次，在测试类还没有实例化就已经被加载，所以用 static 修饰
@AfterClass：所有测试方法执行完执行一次，在测试类还没有实例化就已经被加载，所以用 static 修饰
@Ignore：暂不执行该测试方法

15.4.2 assert

刚刚图中方法内的 assert 关键字是断言，JDK4 新特性，在测试阶段时使用，保证程序的正确性。

语法：assert 布尔值;、assert 布尔值 : "错误信息";

Object result = method();
assert result == 预期结果;

如果布尔值为 false，将会抛出 AssertionError 异常，不再需要 sout 输出，通过人眼验证结果。

在测试类中可以使用 Assert.assertEquals("预期结果", "实际结果"); 代替。

断言默认关闭，可以运行时加入 -enableassertions 或 -ea 参数，启动断言。

打开 Edit Configurations

在 VM options 框内输入 -ea 即可，默认开启所有类的断言，可以使用 -ea:完整类名 只启动某个类中的断言。

如果没有 VM options 框，点击 Modify options，添加 Add VM options。

开启断言后，可以使用 -da:完整类名 或 -disableassertions:完整类名 禁用某个类中的断言。

15.4.3 日志

不是则在方法1多处插入 sout 语句，观察究竟是哪里出了问题，或者 debug；好，发现错误处修改后，再手动将 sout 语句一个个删掉。

这样做比较麻烦，使用日志记录器记录结果，打开记录或关闭都比较方便。

日志级别，从高到低分为如下级别：

  • SEVERE (highest value）：严重
  • WARNING：警告
  • INFO：信息
  • CONFIG：配置
  • FINE：详细
  • FINER：较详细
  • FINEST (lowest value）：非常详细

Logger 类中：

Logger getGlobal()：获取全局记录器

Logger getLogger(String name)：创建给定名称的记录器，一般为包名，若有此名称的记录器，则直接返回

String getName()：获取记录器的名称

void log(Level level, String msg)：记录某级别的消息

void log(Level level, String msg, Throwable thrown)：记录异常信息

void logp(Level level, String sourceClass, String sourceMethod, String msg)：记录某级别的消息，指定所在类和方法名。

void config(String msg)：记录 config 级别的消息

void info(String msg)：记录 info 级别的消息

void severe(String msg)：记录 severe 级别的消息

void setLevel(Level newLevel)：设置日志级别，指定此记录器将记录哪些消息级别。低于此值的消息级别将被丢弃。级别值 Level.OFF 可用于关闭日志记录，级别值 Level.ALL 打开所有级别的日志记录；

如果 newLevel = null 且有父记录器，设置为父记录器的级别；默认为 INFO 级别；

若子记录器没有设置级别，修改父记录器的级别同时会修改子记录器的级别。

void setParent(Logger parent)：设置 parent 为此记录器的父记录器，若此记录器没有设置级别，级别设置为父记录器的级别。

// 默认级别为 info，以下级别的消息不会显示
Logger global = Logger.getGlobal();
// 会显示
global.info("info 级别的消息");
// 不会显示
global.config("config 级别的消息");

// 设置日志级别为 SEVERE，以下级别的消息不会显示
global.setLevel(Level.SEVERE);
// 会显示
global.severe("severe 级别的消息");
// 不会显示
global.info("info 级别的消息2");

你可能发现修改了级别，控制台上却没有显示，打开 JDK 安装目录下的 jre/lib/logging.properties 日志记录配置文件，修改 java.util.logging.ConsoleHandler.level = FINEST

############################################################
# 使用指定的日志记录配置文件
# 运行时添加选项：-Djava.util.logging.config.file=配置文件的绝对路径
############################################################

handlers= java.util.logging.ConsoleHandler
# 默认是控制台处理器，负责将消息打印在控制台上
# 可以换成 java.util.logging.FileHandler，消息会输出在指定文件中
# 多个处理器使用 , 分隔

.level= INFO
# 日志记录器的默认级别为 INFO，低于此级别的消息会被丢弃，代码中可以使用 setLevel 方法设置级别
# 指定日志记录器的默认级别为 INFO，此配置文件添加一句
# 日志记录器名.level=INFO

java.util.logging.FileHandler.pattern = %h/java%u.log
# 若有 FileHandler，会将消息输出到 %h/java%u.log，可以改为其它路径
# 注意不要含有中文，确保目录存在
# %h，用户的主目录，可以通过 System.getProperty("user.home") 获取
# %u，是防止文件重名的编号
# 在 Windows 系统下，此日志文件路径为 C:\Users\用户名\java?.log

java.util.logging.FileHandler.limit = 50000
# 日志文件最大字节数，改 0 代表无限制
# 如果日志文件个数为 1，超过 50 kb 后就会覆盖原有文件内容
# 如果日志文件有多个，超过 50 kb 后，将第一个文件内容移到下一个文件，依次类推
# 最后将最新日志输出到第一个文件中

java.util.logging.FileHandler.count = 1
# 日志文件个数为 1
# 如果只有单个文件，每次运行内容都会被覆盖
# 如果有多个日志文件，每次运行输出内容到第一个文件，第一个文件原有内容转移到第二个文件中
# 依次转移，直到最后一个文件内容被倒数第二个文件内容覆盖

java.util.logging.FileHandler.formatter = java.util.logging.XMLFormatter
# 默认以 xml 格式保存，可以改为 java.util.logging.SimpleFormatter 以普通文本格式保存
# 添加如下属性，每次运行时追加日志到第一个文件尾部
# java.util.logging.FileHandler.append = true

java.util.logging.ConsoleHandler.level = INFO
# 控制台上打印的消息限制在 INFO 以上，低于此级别的消息不会在控制台上输出
java.util.logging.ConsoleHandler.formatter = java.util.logging.SimpleFormatter
# 以普通文本格式输出在控制台上

15.x 总结回顾

运行时异常应以预检查的方式避免，用 try catch 处理，效率太低。

编译时异常不确定如何处理或让调用者处理就 throws 上抛，否则就 try catch 处理。

throws 与 throw 的区别

关键字 作用 位置 所跟 意义
throws 异常处理的一种方式 方法声明处 可能抛出的异常类型 表明自己不处理异常，交给调用者处理
throw 抛出异常对象的关键字 方法体中 异常对象 说明程序某处出现了错误
